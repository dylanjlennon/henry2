### API endpoints
**Status**: All working

| Endpoint | Result |
|----------|--------|
| `POST /api/web/search` | Returns `runId` in ~300ms |
| `GET /api/web/status/:id` | Streams run progress correctly |
| `GET /api/web/history` | Returns last N runs |
| `GET /api/artifact/:id` | Serves blob bytes; returns 405 on HEAD (cosmetic, doesn't affect download links) |

**Known issue — provenance totals always 0 in production:**
`artifactsProduced` and `httpHits` fields in the run summary always report 0. Root cause: in the serverless fan-out model, each fetcher runs in its own isolated function invocation. The orchestrator's `ProvenanceRecorder` never observes individual `putArtifact` or `httpFetch` calls from child functions — those are recorded by the child recorder instances. The aggregate totals on the parent run row stay 0. Only `fetchersCompleted` / `fetchersTotal` are accurate (set via `setFetcherResultTotals` on the results returned to the orchestrator). **Not fixed in this pass** — requires a post-run aggregation query or passing totals back in the fetcher response payload.

## Fetcher findings

### historic-district
**Status**: Consistently failing — data unavailable

`arcgis.ashevillenc.gov` does not resolve from cloud hosts (DNS failure). The fetcher already handles this gracefully: it returns `status: completed` with `layerChecked: "unavailable: fetch failed"` and `inLocalHistoricDistrict: false`. The compliance tile shows "unavailable: fetch failed" in its detail row.

**Root cause**: The City of Asheville appears to have changed or restricted access to this ArcGIS server endpoint. The domain does not resolve (`Could not resolve host: arcgis.ashevillenc.gov`). An alternative Asheville open data endpoint needs to be identified. **Not fixed in this pass** — requires identifying the new Asheville GIS REST URL for the Zoning Overlays layer.

### gis-map
**Status**: Improved — ~20% crash rate reduced to clean failure

The `gis-map` fetcher crashed with `"page.pdf: Target page, context or browser has been closed"` approximately 20% of the time. Root cause: the AbortSignal fires (fetcher timeout) during the PDF generation step after the long map-rendering waits. When this happens, `browser.newPage()` throws because the browser is already closing, and the original catch block attempted `page.pdf()` on the same closed browser — which also threw, causing an unhandled cascade.

**Fix applied** (`src/fetchers/gisMap.ts`):
- The inner `try/finally` block now explicitly closes `htmlPage` on both success and failure
- The outer catch wraps the fallback `page.pdf()` in its own try/catch; if both fail it rethrows with a clear message that propagates to the fetcher's `status: failed` result
- The run will now correctly show `partial` (not crash) when gis-map times out during PDF conversion

The 20% failure rate itself is not eliminated — it's inherent to the long wait times required for the Buncombe County GIS map to fully render. Reducing wait times risks capturing a blank/county-outline-only screenshot.

### simplicity-property
**Status**: Working as designed (skipped for non-Asheville parcels)

Skips properties outside Asheville city limits. The jurisdiction determination uses a different data source than the `jurisdiction` fetcher. Behavior is correct.
